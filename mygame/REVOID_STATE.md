## Project Overview

Re:Void is a text-based MUD built on **Evennia 4.x / Python 3.12**. It is a roleplay-focused game with a strong emphasis on character description, consent, freeform RP tools, and a wisp (OOC spirit) system. The game is inspired by an existing MUD but is being built with more mechanical substance — freeform items are tracked, locks are real, written items persist, etc.

## Key Systems

### Character Description — 16 Layers
`return_appearance` in `typeclasses/characters.py` assembles output in layers. Proximity is auto-detected from both characters' `db.proximity` dicts.

| # | Layer | Condition |
|---|---|---|
| 1 | Name + title | Always |
| 2 | Species + age | Always |
| 3 | Core description | Always |
| 4 | Outfit summary | Always |
| 5 | Body language | Always |
| 6 | Scent | Proximity (near/with) |
| 7 | Outfit details | Deep OR proximity |
| 8 | Presence/aura | Deep examine |
| 9 | Touch description | Deep AND proximity |
| 10 | Markings | Visibility-gated (casual/intimate/mature) |
| 11 | Voice/mood | Always |
| 12 | State tells (restraints, lead) | Always |
| 13 | Bio | Always |
| 14 | Intimate | Proximity + mature consent |
| 15 | Zone details | Deep examine |
| 16 | RP hooks | Deep examine |

### Wisp System
- Wisps are Accounts in OOC mode — NOT in `room.contents`
- Located via `account.db.wisp_location` (room dbref)
- Found via `WispVisibility.get_room_wisps(room)` in `world/wisp_visibility.py`
- Wisp commands use context-aware keys: same `say`/`pose`/`emote` keys as character commands, resolve differently based on AccountCmdSet vs CharacterCmdSet

### Proximity
- Stored on `character.db.proximity` as `{str(other_id): level}` where level is "near" or "with"
- Commands: `approach` (→ near), `beside` (→ with), `withdraw` (→ remove)
- Proximity gates description layers only — does NOT block RP actions
- `aside` uses proximity: room sees gesture, only two people hear content, wisps hear everything

### Consent System
- `character.db.consent_flags`:
  - `casual`, `intimate`, `mature`, `bdsm`, `lead_follow`, `restraint`, `plock`
- Double opt-in: both looker AND subject must have the flag for gated layers to show
- `consent plock on` required before another player can apply a permanent freeform lock

### Freeform System
**Items on characters** (`character.db.freeform_items`):
- Placed with `place <target> <zone> = <name>/<description>`
- Zone must exist on target character
- `slock <target> <name>` — scene lock, auto-releases at scene end, returns code
- `plock <target> <name>` — permanent lock, requires plock consent, creates real Key object in locker's inventory
- `unslock <target> <name> <code>` / `unplock <target> <name>` (need key)

**Sensory layers on rooms** (`room.db.freeform_sensory`):
- Types: sound, scent, light, taste, texture, temp
- `/pin` switch makes them survive scene end
- All cleared (non-pinned) when `FreeformManager.end_scene(room)` is called

**Scene end cleanup** is triggered in `scene_commands._end()` → calls `FreeformManager.end_scene(room)`

### Wardrobe System
- `character.db.wardrobe` — individual named items (zone + desc + examine + ambient)
- `character.db.outfit_presets` — full multi-zone snapshots
- Save anywhere: `wardrobe save <zone> "<name>"` or `freeform save <item> "<label>"`
- Apply: `wardrobe wear "<name>"` / `outfit wear "<name>"`
- Designed for players like Helena who switch between multiple forms (human, futanari, full wolf) each with different zone configurations

### Text Editor
- `world/text_editor.py` — universal multi-line editor
- Enter with any command that opens it (rdesc, rtime, write, etc.)
- `:help` / `:?` — in-editor help with all commands and color codes
- `:save` — save without exiting
- `:q` / `:done` — save and exit
- Extra setter fallback: builder/freeform commands pass `extra={"setter": fn}` to use editor without registering a static key

### Key Objects (`typeclasses/key.py`)
- `target_type`: "zone" or "freeform"
- `target_zone`: zone name (zone locks)
- `target_item`: item name (freeform locks)
- `setup_key(target_char, zone_name, lock_type, creator, target_type, item_name)`
- Looking at a key shows target, zone/item, lock type, who locked it, when

## NPC System

Built and registered. Three new files:

| File | Purpose |
|---|---|
| `typeclasses/npc.py` | NPC typeclass — tier system, ambient pool, trigger/service hooks, builder sheet |
| `world/npc_loader.py` | YAML-driven spawner — update-in-place via `npc_id` tag category |
| `commands/npc_commands.py` | Builder: `CmdNPC`. Player: `CmdExtra`, `CmdGreet`, `CmdAsk`, `CmdNService` |

**Tier system:**
- `NPC_TIER_EXTRA = 0` — scene extras (room.db data, not real objects)
- `NPC_TIER_AMBIENT = 1` — ambient only, no triggers
- `NPC_TIER_SCRIPTED = 2` — full trigger/reaction system
- `NPC_TIER_INTERACTIVE = 3` — service system

**Key integration points:**
- NPC `get_room_desc_line()` → feeds room Layer 6 (presence lines)
- NPC `get_ambient_pool()` → auto-picked up by room ambient script
- NPC `on_hear_say(caller, text)` → parrot mechanic, fires when `react_to_say = True`
- YAML format documented in `world/npcs/README.md`
- Scene extra cleanup wired into `FreeformManager.end_scene()` via try/except import

**Registration:** Both `ALL_NPC_BUILDER_CMDS` and `ALL_NPC_PLAYER_CMDS` added to `commands/default_cmdsets.py`.


```
evennia shell

```

### Reloading NPC YAML After Edits

`load_file` only reads the YAML — it does NOT update live NPCs.
The correct command to apply YAML changes to existing NPCs:

```
@py from world.npc_loader import NPCLoader; [NPCLoader.spawn_from_config(c) for c in NPCLoader.load_file("world/npcs/forming.yaml")]
```

Then `@reload` if code also changed.

### After Running the Build Script

1. Note the dbref table printed at the end
2. Update `world/npcs/forming.yaml` — change all `location: "#2"` to the actual room dbrefs
3. Run `build_all()` again to move NPCs to their correct rooms
4. Once a hub room is built, update the Space 5 exit in `build_forming.py` (search `TODO: change #2`)
